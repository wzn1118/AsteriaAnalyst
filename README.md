# Asteria Analyst

> 本地优先的数据分析与报告交付工作台：从一份结构化数据和一个业务问题，走到可审阅的分析证据、报告资产与修订版本。

[下载 Windows 便携版](https://github.com/wzn1118/AsteriaAnalyst/releases/latest) | [全部可用功能目录](docs/feature-catalog.zh-CN.md) | [完整模块手册](docs/module-guide.zh-CN.md) | [功能逐项参考](docs/function-reference.zh-CN.md) | [Skill、Agent Team 与 Runtime 指南](docs/local-extensions.zh-CN.md) | [快速开始](docs/getting-started.zh-CN.md) | [详细用户指南](docs/user-guide.zh-CN.md)

## 项目定位

Asteria Analyst 面向经营分析、运营复盘、研究统计和咨询交付场景。它不是一个只根据提示词生成结论的写作工具，而是把数据、业务问题、受控分析、证据、报告和后续修改组织成一条本地可追溯的工作流。

一个典型任务从 CSV、Excel、TSV 或 Stata 数据开始：先确认工作表、行粒度、字段类型、缺失、单位与时间范围；再定义业务问题、目标读者和约束；随后选择受控统计分析或 Analysis Lab 方法实验；最后审阅表格、图表、计算证据与报告资产。需要正式管理报告时，系统还会执行强制 AI 审计链、确定性数值计算、证据校验和 PDF 发布门禁。

| 面向角色 | 主要问题 | Asteria Analyst 提供的能力 |
| --- | --- | --- |
| 经营或运营负责人 | 如何把周期性业务问题变成可讨论的决策材料 | 业务简报、受控分析路径、报告资产与审阅状态 |
| 数据分析师或研究人员 | 如何理解字段、选择方法并保留验证依据 | 数据画像、统计分析、Analysis Lab、方法卡和结果工件 |
| 报告或交付负责人 | 如何处理修改而不丢失上下文和证据 | 修订会话、批注、附件、文件差异和本地版本发布 |
| 工程、治理或审计伙伴 | 如何确认结论来自哪里 | 确定性计算、证据绑定、trace 工件、manifest 和公开边界 |

## 从数据到交付

<pre>
数据、工作表与业务问题
  -> 数据画像与字段语义
  -> 业务路由、指标与方法规划
  -> 受控统计或确定性计算
  -> 证据表、图表与报告部件
  -> 质量检查与下载资产
  -> 批注、修订和版本发布
</pre>

这条路径的重点是可复核性：模型可以帮助理解字段、整理业务上下文和规划指标，但正式数值不能由语言模型编造或定稿。图表、表格、方法结果和报告文本应能回到相应的数据与计算证据。

## 六个工作区

| 用户目标 | 页面/路由 | 先准备什么 | 页面内实际操作 | 完成后得到什么 | 状态与下一步 |
| --- | --- | --- | --- | --- | --- |
| 快速组织一项报告任务 | 首页 <code>/</code> | 数据、业务问题、目标读者和可选背景材料 | 上传或选择数据，关联材料，填写需求与交付预期，创建任务 | 与正式分析共用的数据画像、任务进度、报告章节和下载资产 | 需要本机后端；要配置统计或正式报告时继续进入 <code>/analysis</code>。 |
| 做受控统计或生成报告候选 | 正式分析 <code>/analysis</code> | 数据集/工作表；可选目标、特征、分组、时间字段；报告要求 | 选择多表策略，运行受控快速统计，设置报告深度、视觉和交付偏好，提交报告任务 | 统计结果、任务阶段、结构化章节、图表、表格、资产和质量状态 | 基础工作流默认可用；正式 <code>management_report.pdf</code> 仍受后文的强制链与门禁约束。 |
| 探索、比较和复核方法 | Analysis Lab <code>/lab</code> | 数据集、活动表、字段角色、方法参数和可选重点对象 | 定义派生口径，选择方法卡，创建独立运行实例，单独/批量/合并运行 | 方法可运行性、阻断原因、<code>run_id</code>、方法结果、图表、表格、审计和实际返回工件 | 用于探索和证据复核；结果不能绕过正式报告发布门禁。 |
| 先理解方法该如何配置 | 方法指南 <code>/lab/method-guide</code> | 待解决的问题与字段角色的初步判断 | 依次学习重点对象、字段绑定、运行实例和模型输入字段 | 可带回 Lab 的配置步骤、适用条件和注意事项 | 默认可用；它不上传数据、不运行方法，也不生成分析文件。 |
| 查找已有报告并选择要续改的对象 | 修订中心 <code>/revision</code> | 已有本地报告及其索引元数据 | 按关键词、数据集或业务画像筛选和排序，查看详情，创建或恢复会话 | 报告列表、详情、会话记录和进入工作区的入口 | 空库是合法初始状态；需要实际修改时进入 <code>/revision/workspace</code>。 |
| 审阅、批注、续改和登记本地版本 | 修订工作区 <code>/revision/workspace</code> | 一份已有报告、修改意见；可选附件、批注和本机运行时 | 提交修改意见，查看 HTML/PDF，添加批注与附件，比较差异，生成并发布修订版 | 会话事件、预览、差异、附件、产物和本地版本状态 | 高级续改依赖可选运行时；“发布”只表示本地报告版本，不是 GitHub 发布。 |

各工作区的完整功能、对应后端模块和启用条件见 [模块与功能手册](docs/module-guide.zh-CN.md)。

## 能力状态与边界

公开项目把能力明确分为四类，避免用户因按钮或目录而误以为所有能力都无需配置即可运行：

| 状态 | 能力 | 使用条件 |
| --- | --- | --- |
| 默认可用 | 本地数据导入、数据画像、受控统计、智能报告界面、Analysis Lab、方法指南、报告浏览 | 启动本机前后端即可使用；具体分析结果取决于数据质量和方法适配性 |
| 可选配置 | AI 辅助字段语义、业务路由、指标规划、报告与修订增强 | 使用者在本地 <code>.env</code> 中配置兼容的 AI 提供方，并承担相应的数据合规责任 |
| 默认关闭 | Codex Runtime、外部 Skill、本地报告 Agent Team、非沙箱本地执行、Codex 搜索 | 仅可信本机的管理员在明确授权后通过服务端环境变量启用 |
| 受发布门禁约束 | 正式 <code>management_report.pdf</code> | 必须完成 AI trace、schema 校验、确定性数值、证据校验和质量门禁；最终分数低于 90 不得发布 |

公共版不会在浏览器中要求或保存 API Key，也不把任意 Python/R 脚本执行作为普通用户功能。运行时密钥和执行策略由本地服务端配置管理。

## 功能概览：页面、操作、输入、结果与边界

下面的内容是最常用工作流的摘要。所有页面、API 和本地扩展的逐项清单先见 [全部可用功能目录](docs/feature-catalog.zh-CN.md)；本节继续说明常用操作从哪里进入、需要准备什么、成功后能看到什么，以及不能据此作出的过度承诺。需要逐项接口、请求字段和异常状态时，继续阅读[功能逐项参考](docs/function-reference.zh-CN.md)；需要判断一个页面完成后应得到什么，阅读[逐页结果说明](docs/page-results.zh-CN.md)。

### 1. 工作台入口与正式分析：<code>/</code>、<code>/analysis</code>

根路径和 <code>/analysis</code> 共用智能报告工作台。根路径额外展示公开发布版的本机运行时边界；<code>/analysis</code> 保留正式报告生成入口。两者都不是静态演示页，必须连接已启动的后端后才会加载数据、创建任务和返回产物。

| 可以完成的操作 | 需要选择或填写的内容 | 成功后实际看到或得到的结果 | 关键边界 |
| --- | --- | --- | --- |
| 导入或复用数据集 | CSV、TSV、Excel <code>.xlsx</code>、Stata <code>.dta</code> | 数据集标识、文件和工作表信息、行列规模、字段摘要、样本行、缺失提示和当前活动工作表 | 旧版 <code>.xls</code> 必须先转换为 <code>.xlsx</code>；部分浏览器控件可能显示该扩展名，但后端不会将其作为分析数据接收。文件可读取不代表表头、单位、日期、主键或数据粒度正确。 |
| 选择多工作表策略 | 当前工作簿中的工作表，以及单表、合并、分别或组合策略 | 表级数据画像和带入任务的工作表上下文 | 系统不会猜测不同工作表之间的连接键、汇总层级或业务口径；这必须由用户确认。 |
| 关联历史报告和业务背景 | 本机历史报告、背景材料或明确的业务说明 | 可被任务引用的材料记录和上下文 | 历史材料只帮助理解任务、结构或交付风格，不能替代当前原始数据，也不是未经验证事实的来源。 |
| 写出业务简报 | 分析对象、时间范围、要回答的问题、目标读者、核心用途、期望结果、约束条件 | 供路线选择、报告结构和修订会话使用的任务上下文 | “请分析一下”可以创建基础任务，但不能代替可检验的业务问题；简报本身不是数值证据。 |
| 运行受控快速统计 | 当前数据集/工作表，以及目标、特征、分组或时间字段 | 统计表、模型摘要、诊断、图表规格或计算证据，取决于本次方法和后端返回 | 可直接配置相关性、OLS、随机森林、K-Means、PCA、正态性检验等；一次请求最多选择 8 个特征字段，仍须满足样本量和方法前提。 |
| 设置报告任务 | 报告深度、视觉风格和说明、交付目标、全文表格偏好，以及上述数据与业务上下文 | 可执行的报告任务请求 | 这些选项控制交付组织方式，不会把缺失数据、无效字段或不成立的假设自动变成结论。 |
| 跟踪任务和审阅资产 | 已提交的任务 | 阶段名称、进度、阶段事件、结构化章节、关键指标、图表、表格和后端实际返回的下载资产 | 任务完成只表示请求成功；看到报告预览或普通下载不等于正式 <code>management_report.pdf</code> 已通过发布门禁。 |

### 2. 受控统计：从方法目录到可复核结果

统计目录覆盖描述与分布、分组与交叉、相关、回归、组间检验、分类变量检验、非参数与重采样、探索预测、时序与稳定性等类别。目录中的方法名称用于检索、解释和路由；每个方法是否能在当前数据上运行，由字段类型、已选变量、参数和环境共同决定。Lab 会把方法区分为 <code>live</code>、<code>catalog</code>、<code>planned</code> 或 <code>learned</code> 等状态，不能把目录中所有名称写成默认可运行功能。

| 分析类别 | 典型可选方法 | 必需或常见输入 | 结果怎样使用 |
| --- | --- | --- | --- |
| 描述、分布和 KPI | 描述统计、频数、分位数、截尾均值、Winsorize、基尼、帕累托 | 数值或类别字段，必要时排序或阈值 | 获得统计表、分布摘要或排序结果；它描述样本，不自动解释原因。 |
| 分组和交叉 | 交叉表、透视表、分群 KPI | 维度、指标、分组字段 | 获得分组汇总、结构占比和对比表；需要先确认分组口径与样本量。 |
| 关系和模型 | Pearson、Spearman、Kendall、OLS、岭、Lasso、Elastic Net、Logit、Poisson 等 | 适配的目标变量、解释变量、分组或筛选条件 | 获得系数、拟合、显著性和诊断；相关和模型结果均不等于因果结论。 |
| 组间和分类检验 | t 检验、A/B、ANOVA、卡方、Fisher、Mann-Whitney、Kruskal-Wallis、Bootstrap、置换检验 | 指标、分组、配对、协变量或类别字段 | 获得差异、区间、列联表或显著性；仍要检查独立性、样本稀疏、正态性、方差和实验设计。 |
| 探索、预测和时序 | PCA、K-Means、随机森林、基础神经网络、移动平均、ACF/PACF、ADF、Ljung-Box | 多个特征、可选目标或连续时间字段 | 获得成分、分群、特征关系、预测摘要或稳定性提示；探索模型不能直接视为生产模型或经营决策结论。 |

### 3. Analysis Lab 方法实验台：<code>/lab</code>

Lab 用于把一个问题拆成可单独配置、运行和比较的方法实例。它与正式报告工作台共享数据基础，但结果定位为探索、验证和方法复核，不是绕过正式 PDF 门禁的快捷入口。

| Lab 中的具体动作 | 用户需要配置什么 | 页面或工件会给出什么 | 使用边界 |
| --- | --- | --- | --- |
| 准备数据和字段画像 | 数据集、活动工作表，以及需要检查的字段 | 行列和字段画像、候选字段角色、数据质量和可视化提示 | 字段名推断只是辅助；金额、比例、日期、主键和业务维度仍要由使用者确认。 |
| 绑定字段角色 | 目标、分组、标签、时间、特征、重点对象和图形字段 | 当前方法实例的字段绑定和可运行性提示 | 同名方法的不同字段配置会保存在不同运行实例中，避免覆盖；无效字段会显示阻断原因。 |
| 定义派生指标 | 指标名称、计算口径和依赖字段 | 可被后续方法引用的派生指标定义和审阅上下文 | 派生口径需要业务确认，不能借名称掩盖不支持的计算。 |
| 检索并选择方法 | 方法类别、关键词、数据条件和方法卡说明 | 方法卡、适用条件、阻断原因、推荐组合与执行篮 | 方法被列在目录中不等于对当前数据可执行；先看可运行状态和前提。 |
| 创建运行实例 | 选择模式：字段、全量行或重点对象；对应字段与统计参数 | 独立的 <code>run_id</code>、实例配置、方法组合或批量执行计划 | 不同对象、参数和方法组合应使用独立实例，避免把结果混为同一口径。 |
| 自动合并、单方法、批量独立运行或报告试跑 | 一个或多个已配置实例 | 每个实例的运行状态、分析摘要、指标、图表、表格、文本或方法审计；有返回时还会提供可重跑的报告部件蓝图和下载资产 | 批量模式先保留各实例结果再合并；报告试跑只是工作性输出，不能把自动汇总替代为方法审核，也不能宣称已通过正式 PDF 门禁。 |
| 与正式交付对照 | 已准备的 Lab 数据和方法结果 | 可跳转至正式报告生成链路进行试跑或对照 | Lab 结果、方法卡、蓝图或普通下载均不能单独放行正式 <code>management_report.pdf</code>。 |

### 4. 方法指南：<code>/lab/method-guide</code>

方法指南是一个可操作的学习路径，不会直接上传数据、启动统计任务或生成分析文件。它帮助用户在进入 Lab 前完成四个判断：分析的重点对象是什么、字段如何绑定、为什么要拆成独立运行实例、不同模型输入字段承担什么角色。完成后得到的是可带回 Lab 的配置要点和使用注意事项，而不是可下载的统计结果。

### 5. 报告库与修订工作区：<code>/revision</code>、<code>/revision/workspace</code>

| 页面与动作 | 需要的输入 | 成功后的可见结果 | 不应误解为 |
| --- | --- | --- | --- |
| 在 <code>/revision</code> 查找已有报告 | 关键词、数据集、业务画像或列表筛选条件 | 已有报告列表、排序/筛选结果、报告详情和已有修订会话 | 它不会自动生成新的报告，也不会改变原报告内容。 |
| 创建或恢复修订会话 | 已有报告，以及新会话或既有会话标识 | 可继续处理的修订上下文、会话状态和历史事件 | 会话存在不表示本轮修改已经成功生成。 |
| 提交自由意见或快捷修改任务 | 文字修改意见，必要时附带已有报告上下文 | 回合状态、事件流和轮询进度；在可选运行时可继续生成预览或处理结果 | 运行时不可用、任务失败或验收未通过时，页面会显示状态或错误，不会伪造完成。 |
| 审阅报告内容 | HTML/PDF 预览、刷新、新窗口、缩放、翻页、整页和适宽操作 | 当前可访问的预览文件和阅读状态 | 预览文件不自动证明数值、事实或正式发布资格。 |
| 记录批注与附件 | 浏览、画笔、矩形、高亮、橡皮批注；需要时添加附件和说明 | 可与修订说明关联的批注、附件、事件和文件差异 | 批注是修订输入和审阅记录，不能替代数据证据。 |
| 发布修订版 | 可发布的预览、已完成回合和已有验证条件 | 已登记到本地报告工作区的版本、产物和状态 | 此处“发布”只发布本地报告版本；不会推送 GitHub、创建 GitHub Release、部署网站或解除正式 PDF 门禁。 |

### 6. 专项业务路线与实际产物

系统可以根据当前数据特征、业务目标和配置选择专项路线，而不是把所有数据强行套入同一报告模板。可能命中的方向包括电商商品与经营、互联网运营、采购销售、通用深度经营、市场情报、媒体决策、管理会计和独立行业研究。每条路线都需要相应字段和业务口径：例如采购销售路线需要确认采购与销售表的粒度和连接逻辑，互联网运营路线需要足够的事件或周期信息。

无论走哪条路线，运行成功时可能得到报告正文、图表、表格、CSV/XLSX、HTML、Markdown、JSON、README、manifest 或方法级证据文件；具体文件由本次方法、数据、路线和后端返回的交付清单决定。系统不会因为选择了路线名称就保证某一份文件、行业结论或业务建议一定产生。

### 7. 本地扩展：Skill、Feature Trial、Report Agent Team、Codex Runtime 与 R

这些是已经实现的本机扩展，而不是预告功能。它们默认不属于 Windows 便携版“解压即用”的保证范围，因为会读取本机目录、创建受控任务工作区、调用本机 CLI 或依赖本机 R 环境。完整安装步骤、变量和故障处理见[本地扩展指南](docs/local-extensions.zh-CN.md)。

| 扩展能力 | 从哪里操作、需要什么 | 可以完成的具体动作 | 可见结果或工件 | 启用条件与边界 |
| --- | --- | --- | --- | --- |
| 内置/外部 Skill | <code>/lab</code>；仓库内置的 <code>analysis-delivery-workflow</code>，或经审阅的 GitHub 仓库 URL/本机目录 | 查看项目和本机可识别的 Skill；导入、挂载、取消挂载或删除外部 Skill/插件包 | 来源、挂载状态、内嵌 Skill/命令/MCP 元数据；已挂载内容会形成 <code>external_skill_context.json</code> | 当前公开仓库只随源码提供 <code>analysis-delivery-workflow</code>；其他本机 Codex Skill 只有在用户自己的 <code>CODEX_HOME</code> 中存在时才会出现。管理操作需要 <code>ASTERIA_ENABLE_LOCAL_SKILL_INSTALLER=1</code>；导入和挂载不会自动执行包内命令或启动 MCP。 |
| Feature Trial | <code>/lab</code>；已选择数据集、工作表、插件、功能和用户目标 | 用真实数据画像评估某项外部功能是否适合当前任务，再给出下一步配置建议 | <code>trial_id</code>、就绪度和原因、推荐动作、建议的 Lab 参数、<code>trial.json</code>、<code>field_scores.csv</code>、<code>trial_report.md</code> | 需要本地扩展安装器开关；试验是适配性评估，不是任意第三方命令执行器，也不是正式管理结论。 |
| Report Agent Team | <code>/lab</code>；用户自有、已审阅的本机团队目录和 Markdown 角色说明，已选数据集 | 导入、挂载、取消挂载或删除团队；把角色说明和数据上下文交给一个受控 Codex 任务 | 团队/挂载数量、角色摘要、任务 ID、状态、工作区路径，以及实际执行后可用的摘要、差异、文件或错误 | 当前公开仓库不内置 Team，初始列表为空是正常状态。导入/管理需要本地扩展开关；真正运行还需要可用 Runtime、CLI 和受限工作区。后端创建的是一个协调多个角色说明的任务，不是并行云端多智能体服务，也不同于修订工作区的 Report Agent Session。 |
| Codex Runtime 与管线 | Runtime API、修订工作区和团队任务；受限工作区、可用 CLI 与认证 | 健康检查、创建/查询/取消运行或后台任务、查看日志、管理受控进程、运行多阶段管线和查看学习账本 | CLI 路径/版本、<code>run_id</code> 或 <code>job_id</code>、状态、日志、摘要、文件变化、差异、工件或错误 | 真正执行需要 <code>ASTERIA_CODEX_RUNTIME_ENABLED=1</code> 与 <code>ASTERIA_ENABLE_CODEX_RUNTIME_API=1</code>；它是本机受控能力，不应暴露为公网任意命令入口。 |
| R 工作流 | 已存在可用 <code>Rscript</code> 的本机环境 | 发起 R 数据准备、统计、可视化和工件关联流程 | 可能的 CSV、XLSX、Markdown、HTML、PDF 或 R 工件索引 | R 未安装或路径错误时会明确不可用；不会静默替换成另一种计算器。 |

无论是 Skill、Trial、Team、Runtime、R 还是修订 Agent，它们都只能辅助组织分析、任务和材料，不能单独越过正式管理报告的 AI 强制链、确定性数值计算、证据校验和质量门禁。

## 正式管理报告的可信机制

正式 <code>management_report.pdf</code> 必须完整通过以下链路：

<pre>
raw data
-> DataProfileService
-> AIFieldSemanticMapper
-> AIBusinessContextRouter
-> AIMetricDerivationPlanner
-> DeterministicMetricExecutor
-> EvidenceValidator
-> ReportBindingLayer
-> FormalPDFReleaseGate
-> management_report.pdf
</pre>

- <code>AIFieldSemanticMapper</code>、<code>AIBusinessContextRouter</code> 和 <code>AIMetricDerivationPlanner</code> 必须在正式报告绑定前执行，并写入可追溯工件。
- AI 输出必须通过 schema 校验；AI 用于结构化理解和规划，不生成最终数值。
- <code>DeterministicMetricExecutor</code> 负责实际数值计算；<code>EvidenceValidator</code> 校验输入、计算和绑定证据。
- 缺少或无效的 AI trace 时，最多只能生成 <code>debug_report</code> 用于诊断，禁止发布正式 <code>management_report.pdf</code>。
- 最终质量分低于 90 时，同样禁止正式 PDF 发布。

更多细节见 [正式报告可信机制](docs/report-integrity.zh-CN.md)、[AI 强制发布链（中文）](docs/architecture_ai_mandatory_chain.zh-CN.md) 与 [英文原始规范](docs/architecture_ai_mandatory_chain.md)。

## 快速开始

### Windows 便携版

1. 从 [GitHub Releases](https://github.com/wzn1118/AsteriaAnalyst/releases/latest) 下载 <code>AsteriaAnalyst-portable.zip</code>。
2. 完整解压到可写目录，不要在压缩包预览器中直接运行。
3. 双击解压目录中的 <code>start-asteria.bat</code>。
4. 等待启动窗口显示实际本地地址并自动打开工作区。

便携版面向 Windows 本机单用户使用，不是公网托管服务。

### 源码版

前置条件：Python 3.11+、Node.js 20.9+，且首次启动可以连接依赖源。

1. 克隆或解压本仓库。
2. 在仓库根目录双击 <code>start-asteria.cmd</code>。
3. 启动器会检查后端与前端、准备依赖、启动仅绑定 <code>127.0.0.1</code> 的服务、打开 <code>/analysis</code>，并在 <code>logs/launcher/</code> 写入日志。
4. 默认前端地址为 <code>http://127.0.0.1:3000/analysis</code>，后端健康检查为 <code>http://127.0.0.1:8000/health</code>。若端口已占用，启动器会选择附近可用端口并打印实际地址。

基础本地分析不要求 API Key。需要 AI 辅助功能时，从 <code>.env.example</code> 创建本地 <code>.env</code> 并填写自己的兼容提供方配置；绝不要把 <code>.env</code>、密钥、真实数据或生成报告提交到 GitHub。

完整的首次启动、配置和排障说明见 [快速开始](docs/getting-started.zh-CN.md)。

## 推荐使用流程

1. **先核对数据**：上传小型、非敏感样本或 <code>examples/revenue-smoke.csv</code>，确认行粒度、字段、缺失、单位和时间范围。
2. **说明业务问题**：写清希望解释、比较或决策的事项，以及报告读者和约束。
3. **选择正确工作区**：需要探索和比较方法时使用 Lab；需要受控报告路径时使用正式分析；已有报告要续改时进入修订中心。
4. **先审阅证据再读摘要**：依次检查方法/计算结果、表格与图表、文字总结和下载资产。
5. **确认交付等级**：只有全部门禁通过的 <code>management_report.pdf</code> 才是正式管理报告；其他结果是探索性、调试性或修订性材料。

## 数据、隐私与公开边界

- 基础 CSV/Excel 工作可以在本机完成，不要求外部模型提供方。
- 若主动启用 AI 辅助报告或修订，相关字段上下文、报告上下文或上传的工作簿/PDF 可能发送给本地配置的 OpenAI-compatible 提供方。请先脱敏，并遵循数据授权、合同和组织政策。
- 默认部署仅适合回环地址的单用户使用。当前项目没有面向公网多用户的身份认证、租户隔离和生产托管承诺；不要直接暴露到公共网络或共享网络。
- <code>/storage</code> 可提供本地生成工件。公开截图、Issue、演示和提交前都应检查是否含有客户、个人、商业或运行时敏感信息。

非本机部署前，请先阅读 [部署边界](docs/deployment-boundaries.md) 与 [安全策略](SECURITY.md)。
