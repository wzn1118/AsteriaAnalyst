# Asteria Analyst 详细用户指南

本指南按实际工作顺序说明如何使用 Asteria Analyst。核心原则是：先确认数据和问题，再执行分析；先审阅证据，再阅读总结；正式发布只在受控条件满足时进行。

## 1. 先理解两种工作模式

| 模式 | 入口 | 适合什么工作 | 交付要求 |
| --- | --- | --- | --- |
| 正式分析 | `/analysis` | 有明确业务目标、需要受控正式报告的任务 | 通过 AI trace、确定性数值、证据和质量门后交付 |
| Analysis Lab | `/lab` | 选方法、做探索、对比结果、导出方法级资产 | 用于探索与复核；正式交付进入完整发布门禁 |

如果你的问题是“这个数据大致有什么结构、哪种方法合适、有哪些异常值得继续看”，从 Analysis Lab 开始通常更合适。
如果你的问题是“需要形成一份可作为正式管理材料的报告”，从正式分析开始，并准备好数据口径、业务背景和 AI 配置。

## 2. 分析前准备

### 2.1 数据文件

优先准备结构清晰的 CSV 或 XLSX 文件。导入前检查：

- 第一行是否为字段名；
- 一个文件中的多工作表各自代表什么；
- 每一行的业务粒度；
- 日期、金额、数量、比例的单位和格式；
- 缺失值、重复行、合并单元格、说明行和小计行；
- 是否包含个人信息、客户标识、合同数据或其他敏感字段。

对于首次使用，建议先用 `examples/revenue-smoke.csv` 走通流程，再导入真实数据。

### 2.2 业务问题卡

分析效果取决于问题是否清楚。开始前用以下模板写一段简短说明：

```text
目标读者：
业务问题：
分析对象与时间范围：
需要比较的维度：
关键指标及其口径：
希望形成的决策或下一步：
已知限制：
```

示例：

```text
目标读者：月度经营会参与者
业务问题：识别收入变化来自哪里，并确定需要优先验证的业务环节
分析对象与时间范围：按月汇总的区域收入，2025 年 1 月至 12 月
需要比较的维度：区域、月份、产品类别
关键指标及其口径：收入、订单数、客单价；币种为人民币
希望形成的决策或下一步：确定下月复盘需要深挖的区域和产品
已知限制：没有营销投放和库存数据，不能把相关性写成因果
```

这段说明比“帮我分析一下”更能约束范围、方法选择和最终表达。

## 3. 标准分析流程

### 步骤 1：启动并进入正确工作区

使用 [快速开始](getting-started.zh-CN.md) 启动本机服务。通常打开：

```text
http://127.0.0.1:3000/analysis
```

若启动器显示了不同端口，以它打印的地址为准。

### 步骤 2：导入或选择数据集

在正式分析页面选择文件或已有数据集。对于多工作表文件：

1. 先阅读工作表名称；
2. 选择本次问题需要的工作表；
3. 确认所选工作表包含本次分析需要的明细记录；封面、说明页、透视表快照和历史导出通常用于参考；
4. 必要时分别分析不同工作表，不要把粒度不同的数据直接拼在一起。

导入后，给数据集一个能表达用途和时间范围的名称。避免用客户全名、身份证号、邮箱、密钥或合同编号作为公开可见名称。

### 步骤 3：先读数据画像

数据画像是开始分析前的检查点。至少确认：

| 检查项 | 为什么重要 | 常见处理 |
| --- | --- | --- |
| 行数和列数 | 判断数据是否完整、是否选错工作表 | 与源文件或预期范围对照 |
| 行粒度 | 决定分组、去重和指标计算方式 | 在业务问题卡中写清“一行代表什么” |
| 字段类型 | 避免把文本当数字、日期当字符串 | 修正源文件格式或在分析中明确转换 |
| 缺失与异常 | 防止指标被空值、零值或异常值误导 | 记录处理规则，不要静默删除 |
| 时间范围 | 判断趋势、同比或环比是否可比 | 检查月份缺失、跨年度和统计周期 |
| 单位和币种 | 避免千元、元、百分比和小数混算 | 在业务说明和最终材料中显式标明 |

如果字段名含糊、类型混合或时间范围错误，先修正数据或补充说明，再继续。不要依赖模型猜测关键口径。

### 步骤 4：填写业务目标和约束

将“业务问题卡”中的内容写入任务上下文。好的目标应包含：

- 谁需要阅读这份结果；
- 需要回答的一个或两个核心问题；
- 可用的比较维度；
- 必须遵守的指标口径；
- 不应下结论的范围。

避免同时塞入太多没有数据支持的目标。例如没有渠道成本时，不要要求系统直接给出 ROI；可以把它写成“需要补充成本数据后再验证”的后续问题。

### 步骤 5：选择正式分析或进入 Analysis Lab

选择依据如下：

| 你要做的事 | 建议路径 |
| --- | --- |
| 查看描述统计、趋势、分组差异、图表或方法适用性 | Analysis Lab |
| 比较多个方法的结果 | Analysis Lab |
| 需要一条受控、可发布的管理报告链路 | 正式分析 |
| 对已有报告提出修改、查看附件和差异 | 修订中心与修订工作区 |

在不确定方法时，先访问 `/lab/method-guide`。按问题类型、变量类型和方法族阅读适用条件，不要只因为方法名称“高级”就选择它。

### 步骤 6：运行前进行一次口径确认

点击运行前，用以下问题自检：

1. 当前工作表是否正确？
2. 一行数据代表的对象是否清楚？
3. 指标单位、币种和时间范围是否清楚？
4. 是否把不应外发的敏感数据带入了 AI 辅助任务？
5. 是否有足够数据支持所选比较、趋势或分组？
6. 结果是用于探索，还是用于正式材料？

任何一个关键问题不清楚时，停在这一步比运行后返工更省时间。

### 步骤 7：按证据优先顺序审阅结果

结果出现后，不要先只读“结论”。建议按下面顺序审阅：

1. **计算或方法结果**：数值、样本范围、分组、过滤条件和方法是否符合预期。
2. **表格与图表**：趋势、比较、异常和标签是否与数值一致。
3. **证据与限制**：缺失、异常处理、支持范围、假设和无法回答的问题。
4. **文字摘要**：摘要是否准确表达了前面已经确认的证据。
5. **可下载资产**：确认实际文件和交付清单后再分享。

这是避免“文字很像结论，但数据支持不足”的最重要习惯。

## 4. 使用 Analysis Lab

Analysis Lab 用于探索和方法级复核。它通过可执行步骤拆解业务问题，帮助你形成可审阅的方法结果。

### 4.1 选择方法

在 `/lab` 中：

1. 选择数据集和正确工作表；
2. 阅读数据画像；
3. 打开方法指南，按目标和变量类型筛选；
4. 选择一个方法或一组互补方法；
5. 记录你选择该方法的业务原因。

示例思路：

| 问题 | 可先看的方法类型 | 需要注意 |
| --- | --- | --- |
| 哪些类别贡献最大 | 分组汇总、排序、帕累托 | 明确指标、时间范围和类别口径 |
| 最近是否有异常波动 | 趋势、时间序列、异常检查 | 检查数据缺月、节假日和口径变化 |
| 两组表现是否不同 | 分组比较、统计检验 | 确认样本量、分组定义和假设条件 |
| 哪些字段可能相关 | 相关、分层比较、回归类方法 | 相关结果描述变量关系；因果判断还需检查混杂因素与研究设计 |

方法目录中可能同时存在可直接运行的方法和目录级说明。只有在当前数据和环境满足要求时，方法才应被当作可执行选项。

### 4.2 审阅方法级产物

每个方法运行后，优先检查：

- 运行对象、输入字段和筛选范围；
- 结果表是否可解释；
- 图表标题、轴、单位和图例是否正确；
- 方法假设、限制和失败提示；
- 对应的结构化数据和交付清单。

典型方法级资产可能包括 `data.json`、`data.csv`、`data.xlsx`、`preview.png` 和 `README.md`。实际文件以该次任务生成的目录和 manifest 为准。

### 4.3 将探索结果带回业务讨论

Lab 的目标是形成可复核的分析发现和下一步动作。将结果带回业务讨论时，区分三类表述：

- **已验证事实**：本次数据和方法直接支持的数值或趋势；
- **待验证解释**：可能原因、业务假设或需要补充数据的判断；
- **下一步动作**：需要补数据、复查口径、访谈业务方或继续分析的任务。

不要把待验证解释写成已被数据证明的因果结论。

## 5. 生成正式管理报告

正式分析用于需要受控发布的管理材料。它比 Lab 多了一组不可跳过的约束。

### 5.1 启动前要求

正式报告前，确保：

- AI Provider 配置在本机 `.env` 中且可用；
- 业务目标、受众、数据范围和指标口径已明确；
- 数据已经通过必要的字段和质量检查；
- 数据可以按组织政策发送给你配置的模型提供方，或已完成脱敏；
- 你接受正式报告可能因 trace、schema、证据或质量门失败而不发布。

### 5.2 正式链路中各层的职责

```text
raw data
  -> DataProfileService
  -> AIFieldSemanticMapper
  -> AIBusinessContextRouter
  -> AIMetricDerivationPlanner
  -> DeterministicMetricExecutor
  -> EvidenceValidator
  -> ReportBindingLayer
  -> FormalPDFReleaseGate
  -> management_report.pdf
```

| 环节 | 作用 | 用户应检查什么 |
| --- | --- | --- |
| 数据画像 | 描述字段、类型、缺失和结构 | 数据是否选对、粒度是否合理 |
| 字段语义与业务路由 | 解释字段角色、识别业务问题框架 | 关键业务词和口径是否正确 |
| 指标规划 | 区分直接指标、推导指标、假设和不支持项 | 是否避免把假设写成事实 |
| 确定性执行 | 计算最终数值 | 数值、样本、过滤和单位是否可复核 |
| 证据校验与绑定 | 将已验证事实绑定到报告部件 | 图表、表格和摘要是否一致 |
| 发布门禁 | 判断是否可发布正式 PDF | 是否具备 trace、schema、证据和足够质量 |

### 5.3 正式报告没有通过时怎么办

正式门禁未通过时，当前材料进入诊断和复核流程。只有通过全部门禁的结果可作为正式管理报告发布。常见处理顺序：

1. 阅读任务状态、证据问题或质量提示；
2. 回到数据画像和口径，检查选错工作表、字段类型、单位、缺失或时间范围；
3. 核对业务目标是否超出数据支持范围；
4. 检查 AI 配置和 trace 是否有效；
5. 修正问题后重新运行；
6. 只把通过门禁的 `management_report.pdf` 作为正式材料。

在未通过时，系统可以保留 `debug_report` 供诊断。正式报告须通过完整发布门禁后交付。

## 6. 修订中心与版本发布

当分析结果需要多人审阅或后续修改时，进入 `/revision`：

1. 从报告库选择需要续改的报告；
2. 进入 `/revision/workspace`；
3. 用清晰的批注说明“位置、问题、期望变化、依据”，避免只写“这里不对”；
4. 上传授权范围内的补充材料，不要上传密钥、客户隐私或未经授权文件；
5. 查看任务过程、文件和差异；
6. 对照证据和原始业务目标审阅变更；
7. 在版本满足交付要求后再发布。

好的修订意见示例：

```text
位置：管理摘要第 2 条
问题：把区域 A 的环比下降解释为价格变化，但当前数据没有价格字段
期望：改为“观察到收入下降，原因待结合价格和销量数据验证”
依据：本次数据只有月份、区域、收入和订单数
```

## 7. 选择和分享交付物

不同读者需要不同材料。不要默认把所有文件都外发。

| 读者 | 优先材料 | 分享前检查 |
| --- | --- | --- |
| 管理层 | 管理摘要、正式报告、关键图表 | 是否通过正式门禁、结论是否简明且有证据 |
| 业务分析师 | 长版结果、表格、图表、方法说明 | 口径、单位、过滤条件和限制是否可读 |
| 数据分析师 | CSV/XLSX/JSON、方法级资产、manifest | 输入范围、方法版本、复现信息是否齐全 |
| 审阅或治理伙伴 | trace、证据摘要、质量状态、版本差异 | 不包含不应共享的原始数据和密钥 |

实际可下载文件由每次任务的交付清单决定。不要根据文件名推断它已通过正式发布。

## 8. 数据与 AI 使用边界

### 本地基础分析

基础 CSV/Excel 分析可以在本机完成，不需要配置外部模型。即使如此，也应遵循数据最小化原则：只导入完成当前问题所需字段和时间范围。

### AI 辅助分析或修订

当你主动启用 AI 相关能力时，相关上下文或文件可能传输到你配置的 OpenAI-compatible 提供方。使用前确认：

- 该数据能否由该提供方处理；
- 是否需要脱敏、字段删减或抽样；
- 数据保留、审计和跨境传输要求；
- 组织是否允许将工作簿、PDF 或报告上下文外发。

API Key 只应保存在本地 `.env`，不能进入 Git、截图或交付包。

## 9. 提交前检查清单

在发送任何分析材料前逐项确认：

- [ ] 数据集和工作表正确；
- [ ] 行粒度、单位、币种和时间范围明确；
- [ ] 关键指标有可复核计算或表格依据；
- [ ] 图表、表格和文字摘要没有相互矛盾；
- [ ] 推测、假设和因果判断已明确标注；
- [ ] 敏感数据、密钥和内部地址没有出现在导出物中；
- [ ] 若是正式管理报告，AI trace、schema、确定性执行、证据校验和质量门均已通过；
- [ ] 对外分享的文件与接收者角色匹配；
- [ ] 修订版本和发布状态已记录。
