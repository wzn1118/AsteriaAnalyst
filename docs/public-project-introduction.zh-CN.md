# Asteria Analyst 公开项目介绍

> 从一份表格开始，到一组可复核的交付资产结束。

Asteria Analyst 是面向数据团队、运营团队、研究团队与咨询交付团队的本地优先分析交付工作台。它把 CSV、Excel 等结构化业务数据组织为一条可审阅的工作流：理解数据和问题、规划指标与方法、执行确定性计算、沉淀证据和交付物、处理修改意见，并保留可以继续使用的版本上下文。

它的目标是将分析过程沉淀为团队可检查、讨论、交接和复用的材料。

需要逐项查看页面、功能、后端职责模块、启用条件和安全边界时，先阅读 [全部可用功能目录](feature-catalog.zh-CN.md)。它按入口、输入、动作、结果和边界列出公开版全部可用功能；再结合 [模块与功能手册](module-guide.zh-CN.md) 理解完整工作流，或使用 [功能逐项参考](function-reference.zh-CN.md) 核对接口字段。

## 它解决的工作断点

在经营复盘、专题分析和咨询交付中，团队经常遇到同一类断点：

| 工作断点 | 常见后果 | Asteria 的处理方式 |
| --- | --- | --- |
| 数据字段含义不清 | 同一字段被不同人按不同口径理解 | 先生成数据画像，再记录字段语义与分析对象 |
| 业务问题没有转成可执行分析 | 图表很多，但回答不了决策问题 | 将业务背景、受众、指标和方法组织为分析计划 |
| 数值、结论和报告脱节 | 结论无法复核，修改时需要反复返工 | 数值由确定性执行层计算，报告绑定已验证的证据 |
| 交付物只有一份静态文档 | 管理层、分析师和审阅者只能各自重做材料 | 同时产出阅读材料、图表表格、结构化结果和交付清单 |
| 修改意见散落在聊天记录中 | 无法知道哪个版本改了什么、为什么改 | 在修订工作区管理批注、附件、差异和版本发布 |

## 产品承诺

### 输入变资产

用户从数据集、工作表和业务问题开始。系统先帮助确认数据粒度、字段类型、空值、时间范围、单位和候选指标，再进入分析。

### 分析变证据

字段语义、业务路由和指标规划可以使用 AI 辅助，但正式数值由确定性计算层执行。图表、表格、方法结果和结构化输出与相应证据一同保留，方便复核。

### 证据变材料

同一批分析结果可以服务不同阅读者：管理层可阅读决策摘要，分析师可检查方法结果和表格，工程或审计伙伴可查看结构化资产、trace 和 manifest。

### 材料变版本

报告支持持续修订：可进入修订中心，通过批注、附件、过程记录和差异查看完成后续修改，并在审核后发布新版本。

## 从数据到交付的工作流

```mermaid
flowchart LR
    A["数据与业务背景"] --> B["数据画像与字段语义"]
    B --> C["业务路由与指标规划"]
    C --> D["确定性统计与计算"]
    D --> E["证据、图表与报告部件"]
    E --> F["质量检查与下载资产"]
    F --> G["批注、修订与版本发布"]
```

### 1. 数据与业务背景

导入或选择数据集后，明确四件事：

1. **分析对象**：一行数据代表订单、客户、商品、门店、事件还是某个时间周期。
2. **业务问题**：希望解释什么、比较什么、优先做什么决策。
3. **读者和用途**：管理会议、经营复盘、研究分析、客户交付或内部方法验证。
4. **约束条件**：时间粒度、计算口径、敏感字段、历史模板和可用数据范围。

### 2. 数据画像与字段语义

在执行分析前，先检查结构性事实：工作表是否正确、字段类型是否混合、缺失是否影响指标、币种或单位是否一致、时间范围是否足够、主键或分组字段是否可信。

AI 可以辅助解释字段名和候选业务角色。用户在关键口径上完成业务确认，尤其是指标定义、单位、时间窗和数据粒度。

### 3. 业务路由与指标规划

系统将业务背景与数据结构一起用于选择适合的问题框架、指标和方法。规划阶段需要区分：

- 已被数据直接支持的指标；
- 需要从字段推导的指标；
- 只能作为假设或待验证问题的判断；
- 当前数据不支持、不能写成结论的内容。

这一步把“想看什么”变成可解释的分析安排，避免把所有可计算的数字都误写成业务结论。

### 4. 确定性计算与方法执行

关键数值由确定性执行器或选定的方法计算。Analysis Lab 可用于方法探索、比较和方法级输出；正式管理报告使用受控链路，要求完整的 AI trace、schema 校验、证据校验和质量门禁。

### 5. 交付、审阅与续改

成功的工作流可以生成面向阅读、复核和再利用的资产，例如报告内容、图表、表格、CSV/XLSX、HTML、JSON、README、manifest 或方法级证据文件。实际文件取决于所选工作流和本次运行的交付清单。

## 核心工作区：每个页面实际能做什么

| 路由 | 用户在页面上完成的操作 | 要准备的输入 | 成功后会看到或拿到的结果 | 不应误解为 |
| --- | --- | --- | --- | --- |
| <code>/</code> | 选择或上传数据，关联历史报告/业务背景，填写业务问题与交付预期，创建智能报告任务 | 数据文件、工作表、问题、目标读者、用途和约束 | 数据画像入口、任务进度、报告章节、关键指标、图表、表格和后端返回的下载资产 | 静态首页或“只写提示词即可完成分析”；它依赖可运行的后端 API。 |
| <code>/analysis</code> | 在正式分析工作台选择多表策略、配置受控统计、设置报告深度和视觉/交付偏好 | 当前数据集/工作表，必要时目标、特征、分组或时间字段 | 统计结果、任务阶段、报告资产和质量状态 | 所有预览或下载都已经是正式 <code>management_report.pdf</code>；正式 PDF 另受发布门禁约束。 |
| <code>/lab</code> | 给字段分配目标、分组、标签、时间、特征、对象和图形角色；定义派生口径；为方法创建独立运行实例；单独、批量或合并执行 | 已选数据集/活动表、字段绑定、方法参数、可选重点对象 | 方法卡、可运行性和阻断原因、<code>run_id</code>、方法摘要、指标、图表、表格、审计信息与实际返回的工件 | 可以跳过方法假设、数据确认或正式报告门禁的“万能分析页”。 |
| <code>/lab/method-guide</code> | 依次学习重点对象、字段绑定、运行实例和模型输入字段的配置逻辑 | 希望解决的问题和对字段角色的初步判断 | 可带回 Lab 的选择依据、操作要点和注意事项 | 会上传数据、运行方法或自动生成下载文件的分析页。 |
| <code>/revision</code> | 按关键词、数据集或业务画像筛选已有报告，查看详情，创建或恢复修订会话 | 已有报告及筛选/会话条件 | 报告列表、排序筛选结果、详情和可继续的修订上下文 | 自动修改或自动发布原报告。 |
| <code>/revision/workspace</code> | 提交自由修改意见或快捷任务，查看 HTML/PDF，添加附件和批注，比较文件差异并发布本地修订版 | 已有报告、会话、修改说明，必要时批注/附件和可选运行时 | 会话事件、回合状态、预览、批注、附件、差异、产物与本地版本状态 | GitHub 发布、互联网部署，或对正式 PDF 门禁的豁免。 |

### 正式分析具体包含什么

1. **数据接入与工作簿选择**：支持 CSV、TSV、Excel <code>.xlsx</code>、Stata <code>.dta</code>。旧版 Excel <code>.xls</code> 需要先转换为 <code>.xlsx</code>；部分浏览器控件可能显示该扩展名，但后端不将其作为分析数据接收。成功导入后可检查行列规模、字段类型、缺失、样本、工作表和活动工作表。对多表任务，用户必须决定使用单表、合并、分别或组合策略，不能让系统猜测连接键和汇总口径。
2. **业务简报与材料上下文**：可以关联历史报告和业务背景，并填入分析对象、时间范围、核心问题、目标读者、交付用途、期望结果和限制。它们会进入路线选择与报告设计，但不会成为未经验证的事实或数值证据。
3. **受控统计与图表**：可配置描述统计、分组/交叉、相关、回归、组间检验、分类检验、非参数与重采样、PCA、K-Means、随机森林、基础时序等适配方法。结果可能包含数值表、模型摘要、诊断、图表规格和计算证据；目录中的方法名不保证全部可在任意数据上执行。
4. **报告任务与资产**：提交后可查看阶段、进度、结构化章节、关键指标、图表、表格和后端实际返回的文件。实际资产可能是 HTML、Markdown、CSV、XLSX、JSON、manifest、方法证据或报告文件，取决于本次数据、路线和交付清单。

### Analysis Lab 具体包含什么

1. **字段和派生指标**：先读数据画像，再把字段绑定为目标、分组、标签、时间、特征、对象或图形字段；需要时明确派生指标的名称、依赖字段和计算口径。
2. **方法卡和运行实例**：每种方法会展示适配条件、可运行性和阻断原因。用户可按字段、全量行或重点对象创建独立实例，防止同一方法在不同对象和参数下的结果相互覆盖。
3. **执行与复核**：可单独运行、批量运行或把多个结果合并解释。页面显示每个实例的状态和实际返回的摘要、指标、图表、表格、文本、审计信息、下载资产或可重跑的报告部件蓝图。
4. **边界**：Lab 用于探索、验证和方法比较。方法结果、普通报告、Lab 蓝图和下载资产都不能自行升格为正式管理报告。

### 修订工作区具体包含什么

1. **修订输入**：文本修改意见、快捷任务、批注和附件都可以成为会话输入；批注工具包括浏览、画笔、矩形、高亮和橡皮。
2. **过程与结果**：可以查看事件流、轮询状态、HTML/PDF 预览、文件差异、任务状态、附件、产物和已发布本地版本。启用可选运行时后，相关任务还可能返回 CLI 运行、日志、摘要和恢复信息。
3. **本地发布的含义**：修订工作区的发布只把通过当前会话条件的版本登记到本地报告工作区。它不会创建 GitHub Release、推送文件到互联网，也不会改变正式报告必须通过 AI trace、schema、确定性计算、证据校验和质量门禁的规则。

## 适用场景

Asteria 适合结构化数据已经存在、但分析交付仍然分散的场景，例如：

- 月度、季度或专题经营复盘；
- 销售、运营、投放、库存、履约或门店数据诊断；
- 电商、采购销售或多工作表业务数据的结构化分析；
- 研究统计、方法比较和图表复核；
- 咨询交付中的多读者材料准备与后续版本维护。

系统会根据数据质量、口径、样本覆盖和业务背景呈现可支持的分析范围，并将待验证判断明确标记出来。

## 可复核性原则

公开版采用以下分工，避免把语言生成误当成计算或事实认定：

| 事项 | 负责层 | 不应发生的事情 |
| --- | --- | --- |
| 字段含义、业务上下文、分析规划 | AI 辅助层与用户确认 | 直接把模型文本当作最终数值 |
| 数值计算与统计执行 | 确定性执行层 | 用自然语言生成代替计算 |
| 证据与报告绑定 | 证据校验、绑定和渲染层 | 在渲染阶段重新猜测字段或业务关系 |
| 正式发布 | FormalPDFReleaseGate | 在 trace、schema、证据或质量门失败时放行 |

正式管理报告的完整规则见 [报告可信机制](report-integrity.zh-CN.md) 和 [AI 强制发布链](architecture_ai_mandatory_chain.md)。

## 可选本地扩展

公开项目提供 R 工作流、Skill、Feature Trial、Report Agent Team 和 Codex Runtime。这些扩展依赖本机 R、导入本机目录、受控任务工作区或本机 Codex CLI；在可信本机完成相应配置后即可按环境启用。

| 能力 | 入口与前提 | 用户实际操作 | 返回的记录或工件 | 关键边界 |
| --- | --- | --- | --- | --- |
| R 工作流 | 报告工作流；本机必须能解析 <code>Rscript</code> | 启动 R 数据准备、统计、可视化和工件关联 | 可能返回 CSV、XLSX、Markdown、HTML、PDF 或 R 工件索引 | R 不存在或路径错误时会明确不可用；不能把它写成便携版默认保证能力。 |
| Skill | <code>/lab</code>；管理动作需 <code>ASTERIA_ENABLE_LOCAL_SKILL_INSTALLER=1</code> | 查看内置或本机 Skill；从经审阅的 GitHub URL 或本机目录导入、挂载、卸载、删除外部包 | 包来源、挂载状态、内嵌 Skill/命令/MCP 元数据，以及 <code>external_skill_context.json</code> | 随源码提供的项目 Skill 是 <code>analysis-delivery-workflow</code>；本机 Codex Skill 取决于用户环境。导入不会自动执行包内命令或 MCP。 |
| Feature Trial | <code>/lab</code>；已导入插件、已选数据集和本地扩展开关 | 选择插件功能、工作表和目标，用真实数据进行适配性试跑 | <code>trial_id</code>、就绪度、原因、推荐动作、建议参数、<code>trial.json</code>、<code>field_scores.csv</code>、<code>trial_report.md</code> | 它评估字段和数据适配性，不执行任意插件命令，也不直接给出正式业务结论。 |
| Report Agent Team | <code>/lab</code>；已导入/挂载本机 Markdown 团队、已选数据集、可运行 Runtime | 导入团队目录，挂载角色说明，创建一个受控 Codex 团队任务 | 团队和角色摘要、任务 ID、状态、受控工作区路径；实际运行后可能出现日志、摘要、差异、文件或错误 | 当前公开仓库没有内置 Team，初始空列表正常。它创建一个协调角色说明的任务，不提供并行云端多 Agent 调度。 |
| Codex Runtime | Runtime API、修订会话和 Team Run；需 CLI、认证、受限工作区、<code>ASTERIA_CODEX_RUNTIME_ENABLED=1</code> 和 <code>ASTERIA_ENABLE_CODEX_RUNTIME_API=1</code> | 健康检查，创建、查询、取消运行/任务，查看日志、进程、管线和学习账本 | CLI 路径/版本、运行或任务 ID、状态、日志、摘要、文件变化、差异、工件或错误 | 提供本机受控 CLI 任务层，在受限工作区内执行已配置任务。 |

Lab 中的 Report Agent Team 与修订工作区中的 Report Agent Session 是不同功能：前者把团队角色和数据上下文交给新任务，后者围绕已有报告管理消息、附件、批注、事件和差异。两者都不能自行将结果提升为正式 <code>management_report.pdf</code>。完整操作、变量、产物和故障排查见 [本地扩展指南](local-extensions.zh-CN.md)。

## 隐私与公开边界

本项目的 GitHub 公开范围是源码、测试、启动脚本、示例和公开技术文档。下列内容不应进入公开仓库、Issue、截图或演示录屏：

- 客户数据、个人信息、合同限制数据和真实业务报表；
- API Key、访问令牌、内部地址、运行日志和运行会话；
- 未授权的第三方材料；
- 私有作品集、历史交付物或能够反推客户身份的信息。

基础 CSV/Excel 分析可以在本机执行。若用户主动启用 AI 辅助报告或修订，相关字段上下文、报告上下文或上传的工作簿/PDF 可能会发送到用户配置的 OpenAI-compatible 提供方。处理敏感或受监管数据前，应先脱敏并遵循组织政策。

## 不在当前公开承诺内的能力

- 面向公网的多用户 SaaS、身份认证、租户隔离或合规托管；
- 将任何外部模型、运行时、R 环境或扩展技能视为所有安装方式默认可用；
- 将未经公开方法和授权证明的准确率、节省时间、客户成果或规模数字作为产品保证；
- 用 Analysis Lab 绕过正式报告的证据和质量门禁。
