# Asteria Analyst 快速开始

本指南帮助你在 Windows 本机启动 Asteria Analyst，并完成第一次可复核的分析。默认使用本地回环地址；共享或公开部署按独立的架构与安全方案实施。

## 先选启动方式

| 方式 | 适合谁 | 需要安装什么 | 启动文件 |
| --- | --- | --- | --- |
| 源码版 | 开发、二次开发、已经有开发环境的用户 | Python 3.11+、Node.js 20.9+ | `start-asteria.cmd` |
| Windows 便携版 | 希望直接使用随附运行时的 Windows 用户 | 发布包随附运行时 | `start-asteria.bat` |

便携版以 `AsteriaAnalyst-portable.zip` 形式作为 GitHub Release 资产提供。创建 Release 后可从对应页面下载；源码仓库提供源码启动方式。

## 使用前检查

在导入任何业务数据前，先确认以下事项：

- 你有权在本机处理这份数据。
- 启用 AI 功能前完成数据分级与脱敏，仅发送已获授权的字段上下文。
- 你知道每一行数据的粒度，例如订单、客户、商品、门店、事件或日期。
- 你知道数值字段的单位、币种和时间口径。
- 你的首个练习文件足够小、字段名清晰，便于先确认流程。

仓库提供 `examples/revenue-smoke.csv` 作为首次演练样本。真实经营结论应使用当前业务数据、明确口径和完整证据链形成。

## 源码版启动

### 1. 准备运行环境

安装：

- Python 3.11 或更高版本；
- Node.js 20.9 或更高版本；
- 首次启动时可访问依赖源的网络连接。

在 PowerShell 中可选地确认版本：

```powershell
python --version
node --version
npm --version
```

如果系统同时安装了多个 Python 版本，启动器会优先寻找满足 3.11 要求的 Python。已有虚拟环境版本不满足要求时，按启动器的提示重建 `backend/.venv`。

### 2. 双击启动器

在仓库根目录双击：

```text
start-asteria.cmd
```

启动器会按顺序完成：

1. 读取根目录可选的 `.env`；
2. 检查或创建 `backend/.venv`；
3. 安装后端依赖；
4. 检查或安装前端依赖；
5. 构建可用的前端生产包；构建未完成时使用真实的 Next.js 开发服务；
6. 启动只绑定到 `127.0.0.1` 的后端和前端；
7. 对健康地址进行检查后，打开 `/analysis`。

第一次启动耗时通常长于后续启动，因为需要创建环境、安装依赖和构建前端。保持启动窗口打开，直到依赖安装和健康检查完成。

### 3. 记住实际地址

默认地址是：

| 服务 | 默认地址 | 用途 |
| --- | --- | --- |
| 前端 | `http://127.0.0.1:3000/analysis` | Asteria 分析界面 |
| 后端健康检查 | `http://127.0.0.1:8000/health` | 确认 API 已启动 |

若 8000 或 3000 正在被其他应用使用，启动器会在默认端口附近选择空闲端口，并在启动输出中显示实际前端地址和后端地址。以输出中的地址为准。

启动日志在：

```text
logs/launcher/
```

## Windows 便携版启动

### 1. 获取并解压

从 GitHub Release 下载对应版本的 `AsteriaAnalyst-portable.zip`，完整解压到可写目录后运行启动器。

### 2. 启动

在解压目录中双击：

```text
start-asteria.bat
```

便携包带有自己的运行时，并会打开本地工作区。首次启动仍可能需要准备本地缓存或检查资源；请等待窗口显示可访问的本地地址。

### 3. 便携版边界

- 便携版面向 Windows 本机单用户使用。
- 当前启动方式面向本机回环地址；共享或公网部署需要采用独立的部署架构与安全设计。
- 便携版中的数据、设置和运行结果应由使用者自行备份和管理。
- 将便携版升级到新版本前，先导出需要保留的结果和配置，并阅读新版本发布说明。

## 配置 AI 辅助功能

基础数据查看和本地分析可直接运行。AI 辅助的字段语义、业务路由、指标规划、报告和修订功能使用用户配置的 OpenAI-compatible 提供方。

### 1. 创建本地配置

在仓库根目录执行：

```powershell
Copy-Item .env.example .env
```

打开 `.env`，至少填写：

```text
OPENAI_API_KEY=your-key-here
```

按提供方要求设置 `OPENAI_BASE_URL`、`OPENAI_MODEL` 和其他示例变量。密钥保留在本机环境变量或 `.env`，GitHub Issue、截图、报告附件和提交中仅记录掩码信息。

### 2. 理解数据传输

启用 AI 辅助报告或修订后，系统会按任务需要向你配置的 OpenAI-compatible 提供方发送字段上下文、报告上下文或上传的工作簿/PDF。启用前确认数据授权、保留政策和组织合规要求，并为敏感数据准备脱敏版本。

## 第一次分析

1. 打开本地 Asteria 界面，进入 `/analysis`。
2. 选择 `examples/revenue-smoke.csv`，或上传一个小型、非敏感 CSV/XLSX 文件。
3. 如果文件包含多个工作表，选择正确的工作表。
4. 阅读数据画像，确认字段类型、缺失、行粒度、时间和单位。
5. 用一句话写清业务问题，例如“比较不同时间段的收入变化，并找出需要进一步核查的异常”。
6. 根据目的选择：
   - 希望试验方法、比较结果或查看方法级资产：进入 `/lab`。
   - 需要受控的正式管理报告：留在 `/analysis` 并满足 AI、证据和质量门要求。
7. 运行后先检查表格、图表、方法结果或计算证据，再阅读摘要文字。
8. 仅在结果、口径和证据都已审阅后，下载或分享交付物。

完整操作顺序见 [详细用户指南](user-guide.zh-CN.md)。

## 常见问题

### 提示找不到 Python 或版本太低

安装 Python 3.11 或更高版本后重新打开启动器。已有旧 `backend/.venv` 时，按启动器提示重建，并让解释器版本保持一致。

### 提示找不到 Node.js 或 npm

安装 Node.js 20.9 或更高版本，关闭并重新打开终端或资源管理器，然后重试。Node.js 安装不完整时，`node` 存在但 `npm` 可能不可用，应重新安装官方发行版。

### 首次启动卡在安装依赖

确认网络、企业代理、证书策略和包源可用。查看 `logs/launcher/` 中的后端或前端日志，并在 Issue 中使用已脱敏的错误摘要。

### 浏览器打开了，但页面无法连接分析服务

先访问启动输出中的后端健康地址。健康检查异常时查看 `logs/launcher/` 下的 `backend-*.err.log`；后端已就绪而前端仍待启动时查看 `frontend-*.err.log`。以启动器打印的实际端口为准。

### 端口被占用

默认启动会尝试选择附近的可用端口。冲突端口归属会显示在启动日志中；请关闭或重新配置已识别的冲突应用后再启动。

### AI 功能不可用或报告无法通过正式发布

检查 `.env` 中的提供方配置、网络可达性、数据口径和任务提示。正式报告需要有效的 AI trace、schema 校验、确定性数值、证据校验和质量门；API Key 用于提供方连接，发布资格由完整链路确认。详见 [报告可信机制](report-integrity.zh-CN.md)。

### 多人部署

当前启动器服务于 Windows 本机和回环地址。公开或多用户服务通过独立的认证、授权、数据隔离、上传控制、审计和安全设计部署。详见 [部署边界](deployment-boundaries.md)。
