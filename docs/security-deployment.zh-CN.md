# Asteria Analyst 安全与部署说明

本文件是 [SECURITY.md](../SECURITY.md) 与 [部署边界](deployment-boundaries.md) 的中文技术说明，面向本机使用者、维护者和准备发布 GitHub Release 的项目所有者。若英文安全政策与本文件出现冲突，应先核对当前代码、启动器和安全测试，再修正文档或实现；不要扩大安全承诺。

## 1. 当前支持的使用模型

当前公开版本设计为：

```text
一名受信任的本机使用者
    + Windows 本机文件
    + 127.0.0.1 回环地址
    + 本机存储和可选 AI 提供方
```

当前公开版本提供本机单用户工具。账户认证、组织管理、租户隔离、对象级授权、私有对象存储、统一审计、限流、生产备份和事件响应由托管服务平台提供。任何公网部署均需完成相应的安全架构与运维设计。

## 2. 网络边界

| 事实 | 含义 |
| --- | --- |
| 源码启动器默认使用 `127.0.0.1` | 正常情况下只有同一台机器可访问前端和后端。 |
| 便携版默认在本机端口启动 | 启动脚本会从 `8787` 开始寻找可用本机端口，并打开 `/analysis`。 |
| CORS 默认限定本机 Origin | 用于浏览器跨域匹配；用户登录、认证和授权由独立机制提供。 |
| 当前 API 没有身份认证层 | 任何可访问监听端口的人都可能访问相应 API；这正是必须保持回环监听的原因。 |

以下操作都会把本机工具变成需要重新设计和安全审查的系统：

- 将监听地址改为 `0.0.0.0`、局域网地址或公网地址；
- 开放防火墙端口、端口转发、内网穿透或隧道；
- 在反向代理后向多人提供页面或 API；
- 仅通过放宽 CORS 来“允许他人使用”；
- 把 `workspace/storage`、`%APPDATA%\AsteriaAnalyst` 或 `/storage` 映射为共享文件目录。

若确实要建设共享/公网版本，至少需要独立完成：身份认证、会话与账户生命周期、每用户/每租户授权、私有工件下载、数据隔离、上传控制、恶意文件策略、CSRF/安全头、速率限制、审计日志、密钥托管、AI 数据处理协议、任务隔离和应急响应。

## 3. 数据、工件和存储

| 位置或概念 | 安全分类 | 处理要求 |
| --- | --- | --- |
| 上传的数据集、历史报告、业务背景 | 私有本地输入 | 不提交 Git，不贴入公开 Issue，不当作静态资源。 |
| `settings.json`、运行任务、Codex 日志、学习账本 | 私有运行状态 | 保持在私有运行目录，不挂载到 Web 静态服务。 |
| `public_artifacts/reports` 与 `/storage/**` | 本机可引用工件 | 只放明确允许界面访问的产物；仍可能包含敏感报告内容。 |
| `workspace/storage`（源码）或 `%APPDATA%\AsteriaAnalyst`（便携） | 用户数据根目录 | 应位于当前用户可写、非共享、受系统访问控制保护的位置。 |
| GitHub 仓库与 Release 资产 | 公开范围 | 只能包含源码、测试、启动脚本、无敏感示例、文档和经检查的便携包。 |

`main.py` 将 `PUBLIC_ARTIFACTS_DIR` 挂载到 `/storage`，供本机界面引用已选工件；完整存储根目录保持私有。发布截图、日志和工件前逐项检查数据与隐私。

## 4. AI 提供方数据边界

基础文件查看和本地统计可在本机执行，不必配置 API Key。用户主动配置并使用 AI 辅助字段语义、业务路由、报告或修订后，相关字段上下文、业务背景、报告上下文或工作簿/PDF 内容可能发送到其配置的 OpenAI-compatible 提供方。

启用前至少确认：

1. 你有权让该提供方处理该数据类别；
2. 个人信息、客户信息、财务、健康、合同或受监管字段是否需要删除、掩码、聚合或抽样；
3. 提供方的数据区域、保留、访问、审计和合同条款是否满足组织要求；
4. API Key 用于授权调用；数据传输仍需满足安全与合规要求；
5. 报告里的数值和结论仍需依据确定性计算与证据，不可因 AI 文本“看起来合理”而跳过审阅。

启用 AI 后，公开说明应如实描述数据可能发送到已配置提供方的范围。

## 5. 密钥与配置保护

| 建议做法 | 避免事项 |
| --- | --- |
| 在本机 `.env`、进程环境或受保护的系统密钥管理中保存密钥。 | 密钥仅保存在受保护的本机位置，避免写入 `README`、前端配置、浏览器存储、代码常量或截图。 |
| 使用 `.env.example` 作为无密钥模板。 | 将带有真实值的 `.env` 留在私有运行环境。 |
| 通过 `/api/runtime-settings` 的掩码摘要核对是否已配置。 | 将 API Key 保留在受保护的配置中。 |
| 为最小必要的 Runtime 工作区设置路径。 | 将 Codex 与外部 Skill 的可访问范围限制在该工作区。 |

完整变量说明见 [配置参考](configuration-reference.zh-CN.md)。

## 6. 高权限本机扩展

| 功能 | 默认状态 | 必要开关 | 风险与规定 |
| --- | --- | --- |
| 外部 Skill/Agent Team 导入、挂载、运行 | 关闭 | `ASTERIA_ENABLE_LOCAL_SKILL_INSTALLER=1` | 导入或安装不受信任内容会扩大执行面；仅限本机管理员。 |
| Codex Runtime API | 关闭 | `ASTERIA_ENABLE_CODEX_RUNTIME_API=1` | 路径、日志、会话和运行产物不应给共享用户访问。 |
| 非沙箱 Codex Runtime | 关闭 | `ASTERIA_ALLOW_UNSANDBOXED_CODEX_RUNTIME=1` 和相应请求/偏好 | 高风险管理员操作，仅限可信本机按任务需要启用。 |
| HTTP 任意代码执行 | 未作为公开路由发布 | 无 | 当前公开 `main.py` 保持 `/api/code/execute` 未注册状态；新增执行路由需通过独立安全评审。 |

这些开关管理本机高权限能力的启用状态；认证、授权与租户隔离需由独立机制提供。将它们设为 `true` 表示当前机器的操作者明确选择接受相应风险。

## 7. 报告、HTML 与正式发布

- 报告 HTML 预览在隔离语境下查看，并按普通文件内容进行审阅。
- Analysis Lab、自动分析、调试报告、图表和文件下载用于探索、诊断与复核；正式管理报告需通过完整发布链。
- 正式 `management_report.pdf` 必须经过 AI 字段语义映射、业务路由、指标规划、确定性数值执行、证据校验、报告绑定和 `FormalPDFReleaseGate`。
- 正式发布条件包括 trace、schema 校验、确定性数值、证据校验、门禁放行和最终质量分至少为 `90`；未满足条件的结果用于调试或继续复核。

详情见 [正式报告可置信机制](report-integrity.zh-CN.md)。

## 8. GitHub 公开发布卫生

在 push、创建 Release、上传 ZIP、发截图或提交 Issue 前，确认不存在：

- `.env`、API Key、Token、密码、私有 Base URL；
- 客户数据、上传工作簿、报告、修订会话、附件、日志、缓存或本机运行状态；
- 未获授权的作品集、第三方素材或商业交付物；
- 显示用户名、绝对路径、内部域名或可识别业务信息的截图；
- 把“本机可运行”描述成“已通过公网安全审计”的发布文案。

许可证由项目所有者选择并发布。在 `LICENSE` 发布前，复制、修改和商用权利尚未授予；发布脚本和文档仅执行已确定的许可证策略。

## 9. 漏洞报告与安全事件

不要在公开 Issue 中提交漏洞利用步骤、凭据、真实数据、报告、截图或客户信息。使用仓库所有者已经配置且可验证的私密安全通道；若仓库尚未显示私密报告渠道，应先通过项目所有者公开列出的私密联系方式确认通道，再发送最小化、可安全复现的材料。

一份合格的私密报告应包含：受影响版本/提交、影响范围、最小复现步骤、预期与实际行为、安全影响和不含真实数据的安全 PoC。维护者收到报告后应先确认接收、评估影响、制定修复与披露计划；不要为了快速响应而在公开讨论中泄露攻击细节。

## 10. 本机安全检查单

1. 服务是否仍只监听 `127.0.0.1`？
2. 上传的数据和生成工件是否存放在私有本机目录？
3. `.env`、日志、截图和 Git 状态中是否没有密钥？
4. 启用 AI 前是否完成数据授权和脱敏判断？
5. 高权限 Runtime、Skill 或非沙箱执行是否仍保持关闭，除非有明确本机管理员需求？
6. 是否没有把 `/storage` 或数据目录作为共享文件服务？
7. 是否把正式 PDF 与实验/调试输出清楚区分？

需要具体接口行为时参阅 [API 参考](api-reference.zh-CN.md)；准备发布时参阅 [发布与运维指南](release-operations.zh-CN.md)。
