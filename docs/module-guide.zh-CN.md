# Asteria Analyst 模块与功能手册

> 适用于公开仓库和 Windows 本地发布版。本手册说明当前产品模块的用途、入口、输入、输出、启用条件和使用边界，并按规划、默认关闭和可选扩展等状态呈现相关能力。

## 1. 阅读方式与能力状态

### 1.1 本手册覆盖范围

本手册按两层组织：

1. **用户可见模块**：首页、正式分析、Analysis Lab、方法指南、报告库与修订工作区。
2. **支撑职责模块**：数据接入、数据画像、统计执行、报告编排、AI 强制链、资产管理、可选运行时和部署安全。

同一项用户操作通常会跨越多个职责模块。例如，“生成正式报告”会同时使用数据画像、字段语义、业务路由、指标规划、确定性计算、证据校验、报告绑定和 PDF 门禁；页面操作按完整后端步骤执行。

### 1.2 状态标记

| 标记 | 含义 | 典型示例 |
| --- | --- | --- |
| 默认可用 | 在本机成功启动项目后即可使用，不需要外部模型密钥 | 数据导入、数据画像、受控统计、Analysis Lab、报告浏览 |
| 可选配置 | 需要用户在本机配置依赖或 AI 提供方 | AI 辅助报告、AI 修订增强 |
| 默认关闭 | 存在于项目中，但必须由可信本机的管理员明确启用 | Codex Runtime、外部 Skill、本地 Agent Team、非沙箱执行 |
| 受门禁约束 | 只有全部条件满足后才允许得到正式产物 | 正式 <code>management_report.pdf</code> |
| 路由选择 | 仅在数据和业务条件匹配时进入的专项能力 | 电商、互联网运营、采购销售、行业研究路线 |

### 1.3 三条必须先理解的边界

- **数据导入后的检查。** 用户仍应确认每行记录的粒度、字段单位、时间口径、缺失原因、主键或分组逻辑。
- **正式管理报告的发布条件。** 正式 <code>management_report.pdf</code> 必须通过完整 AI 强制链和质量门禁。
- **本地运行时范围。** 默认部署是 <code>localhost/127.0.0.1</code> 的单用户工作台；公网或共享网络部署需配置认证、隔离和网关。

## 3. 模块 01：数据接入与数据目录

### 解决的问题

把分散的结构化数据、历史报告和业务背景放到同一个可审阅的任务上下文中，同时保留数据集、工作表和材料之间的关联。

### 入口

- 首页 <code>/</code> 的智能报告工作台。
- 正式分析页 <code>/analysis</code>。
- Analysis Lab <code>/lab</code>。

### 输入

| 输入类型 | 支持内容 | 用途 |
| --- | --- | --- |
| 分析数据 | CSV、TSV、Excel <code>.xlsx</code>、Stata <code>.dta</code>；旧版 <code>.xls</code> 请先转换为 <code>.xlsx</code> | 作为统计、图表和报告的原始分析数据；部分控件可能显示 <code>.xls</code>，但后端不会将其作为分析数据接收 |
| 历史报告 | <code>.md</code>、<code>.txt</code>、<code>.html</code>、<code>.htm</code>、<code>.pdf</code>、<code>.docx</code> | 提取既有材料的结构或汇报风格，辅助交付设计 |
| 业务背景 | 文本、HTML、PDF、Word，以及部分结构化表格文件 | 提供策略、渠道规则、经营约束、研究目标或使用语境 |

### 核心操作

1. 上传新文件或从已经保存的数据集中选择。
2. 对 Excel 工作簿选择一个或多个工作表。
3. 在不同工作表之间切换，并重新查看其数据画像。
4. 选择或上传历史材料、业务背景材料；必要时粘贴文本内容。
5. 将数据集、工作表、业务目标和材料上下文共同提交给后续分析任务。

### 输出

- 数据集标识、工作表清单和活动工作表。
- 供界面展示的数据摘要和样本预览。
- 与报告任务关联的历史材料和业务背景上下文。

### 前提与状态

基础导入为默认可用能力。首次使用时应先用小型、非敏感样本验证文件结构和字段编码，再处理真实业务数据。

### 使用边界

- 文件上传成功后，字段口径、日期格式、币种、空值和多工作表关系仍需核验。
- 历史报告和业务背景提供上下文；原始事实和正式结论由当前数据、确定性计算和发布门禁确定。
- 上传内容会保存在本地工作区；公开提交、截图、Issue 或演示前必须移除真实客户、个人和商业敏感信息。

### 对应职责模块

后端主要由 <code>dataset_service.py</code> 和 <code>dataset_catalog_service.py</code> 承担；相关 API 包括 <code>/api/datasets</code>、<code>/api/datasets/upload</code>、<code>/api/datasets/{dataset_id}/sheet</code>、<code>/api/historical-reports</code> 与 <code>/api/business-backgrounds</code>。

## 4. 模块 02：数据画像、字段理解与路线建议

### 解决的问题

在运行统计或写报告前，先回答“数据是什么、能做什么、哪些地方需要人工确认”，减少因字段误解、样本粒度混乱或数据质量问题带来的错误结论。

### 入口

- 数据导入后的正式分析数据概览。
- Analysis Lab 的字段流程和字段画像区。
- 正式报告链的 <code>DataProfileService</code>。

### 输入

- 已选择的数据集和工作表。
- 用户确认的目标、分组、标签、时间、特征或对象字段。
- 可选的业务背景与历史材料。

### 核心操作

- 识别行列规模、数值/类别/日期字段、缺失情况、字段摘要和样本行。
- 对多工作表识别表级结构，并支持选择具体工作表。
- 为可视化和分析提供候选字段、数据类型和推荐方向。
- 在 Lab 中把字段指定为结果、解释、分组、对象、时间、图形 X/Y/大小等角色。
- 在需要正式报告时，AI 可以辅助产生字段语义映射；这一步必须留下结构化 trace 和相应的自然语言说明。

### 输出

- 数据画像、字段清单、样本行、缺失与类型摘要。
- 候选字段角色、方法建议和后续分析所需的结构化输入。
- 正式报告路径所需的字段语义映射 trace。

### 前提与状态

基础数据画像默认可用。AI 字段语义映射属于正式报告链的必经步骤，因而需要可用的 AI 配置和有效的结构化输出。

### 使用边界

- 字段名推断、候选主键和表关系推断会受到命名质量、空值、编码和表结构影响，必须由了解业务的人确认。
- “候选角色”供业务人员确认，字段的最终认定以业务定义和数据核验为准。
- PDF 或 HTML 渲染器呈现已确认的字段含义和业务角色；字段理解与业务路由由前置服务完成。

## 5. 模块 03：业务简报、历史材料与交付设计

### 解决的问题

把“请分析这份数据”变成可执行的工作任务：明确要回答的问题、读者、用途、交付形式和限制，形成可支持业务决策的分析结论。

### 入口

- 首页 <code>/</code>。
- 正式分析页 <code>/analysis</code>。

### 输入

- 报告要回答的问题、待解决问题、目标读者、核心用途、期望结果和关键约束。
- 历史报告和业务背景材料。
- 报告深度、高管摘要偏好、视觉风格预设、视觉说明、交付目标和全文表格偏好。
- 已选数据集、工作表和多表组织方式。

### 核心操作

1. 写出业务问题和预期使用场景。
2. 选择单表、合并、分别或组合的多表组织方式。
3. 提供既有材料，以便参考结构或表达风格。
4. 选择适当的交付深度和视觉要求。
5. 将任务交给智能报告编排服务，并在任务进度中审阅结果。

### 输出

- 业务目标、读者、约束和交付偏好组成的任务简报。
- 报告路线、章节或交付部件的规划输入。
- 历史风格适配信息（仅在本次任务实际产生时出现）。

### 前提与状态

填写业务简报本身默认可用。若选择 AI 辅助字段理解、业务路由、指标规划、文本增强或修订，需要在本机配置兼容的 AI 提供方。

### 使用边界

- 历史材料用于增强上下文；正式结论采用当前数据、确定性计算和发布门禁确认的事实。
- 视觉风格会影响交付表达，不改变统计有效性、数据事实或正式门禁。
- 配置 AI 后，相关字段上下文、报告上下文和上传文档可能发送到用户配置的 OpenAI-compatible 提供方；处理敏感数据前必须脱敏并遵守组织政策。

## 6. 模块 04：受控统计、图表与计算证据

### 解决的问题

把分析从“看起来合理的叙述”变成有方法、输入、结果和适用条件的计算证据，并为报告提供表格、图表和可复核的数值来源。

### 入口

- <code>/analysis</code> 中的受控快速统计。
- <code>/lab</code> 中的方法目录、运行实例和方法卡。

### 输入

- 已选数据集、工作表与字段角色。
- 方法所需的结果字段、解释字段、分组字段、对象范围、时间字段或特征字段。
- 方法相关选项，例如显著性、假设方向、聚类数、实验指标或筛选范围。

### 核心操作

| 分析族 | 已验证的代表能力 | 使用时应检查的条件 |
| --- | --- | --- |
| 描述与经营概览 | 描述统计、频数、交叉表、透视表、分位数、截尾处理、基尼、帕累托、KPI | 字段类型、缺失、单位、汇总层级 |
| 关系与关联 | Pearson、Spearman、Kendall、偏相关、距离相关、点二列、Eta squared 等 | 连续/类别变量适配、样本量、异常值 |
| 回归与模型 | OLS、岭回归、Lasso、Elastic Net、稳健回归、分位数回归、Logit、Poisson 和部分诊断 | 因变量类型、共线性、残差和模型假设 |
| 比较与检验 | t 检验、配对检验、Z 检验、A/B 测试、ANOVA、ANCOVA、Welch ANOVA、事后比较 | 分组定义、样本独立性、方差与分布假设 |
| 分类与非参数 | 卡方、Fisher、McNemar、Cochran Q、CMH、Kappa、Mann-Whitney、Wilcoxon、Kruskal-Wallis、Friedman | 列联表条件、配对关系、排序尺度 |
| 探索与时序 | PCA、K-Means、随机森林、基础神经网络、正态性、移动平均、ACF/PACF、Ljung-Box、ADF | 特征预处理、时间顺序、样本覆盖、解释目标 |

前端快速统计提供相关矩阵、OLS、随机森林、K-Means、PCA 和正态性检验等受控入口。Analysis Lab 的方法目录可用于搜索、筛选、查看所需字段与输出类型，并以运行实例保存不同的字段绑定和配置。

### 输出

- 指标、统计量、模型或检验结果。
- 图表、表格、计算表、方法摘要和分析证据。
- 在支持的工作流中，结果可以进一步成为报告部件、资产清单或可重跑的蓝图。

### 前提与状态

受控统计是默认可用能力，但每个方法仍需满足其数据类型、样本量和输入要求。具体能否成功运行、产生何种图表或导出格式，取决于本次数据与方法配置。

### 使用边界

- 公共版通过受控方法提供 Python/R 相关分析路径；普通用户不提供任意 Python/R 代码执行入口。应选择受控方法，并将不可信脚本排除在分析路径之外。
- 项目内的方法目录用于发现和路由，目录条目不保证每一项都已在当前发布版实现为可执行引擎。
- 统计显著、变量相关或模型拟合良好时，因果关系、业务价值和可复制性仍需结合研究设计与业务证据评估。
- 正式报告所用最终数值必须由确定性执行器计算；模型生成内容提供文字说明和结构化建议。

### 对应职责模块

后端主要由 <code>analysis_service.py</code> 和 <code>statistical_catalog.py</code> 承担；主要接口包括 <code>/api/statistics/catalog</code> 和 <code>/api/statistics/run</code>。

## 7. 模块 05：智能报告、任务编排与多表策略

### 解决的问题

将数据、业务目标、工作表组织方式、分析方法和交付设计组合为可跟踪的报告任务，保留每次生成的输入、过程和交付设计。

### 入口

- 首页 <code>/</code>。
- 正式分析页 <code>/analysis</code>。

### 输入

- 数据集、选定工作表与多表策略。
- 业务简报、历史材料、业务背景和交付设计。
- 可选的受控统计结果或分析路线。

### 核心操作

1. 选择单表、组合、分别或合并的工作表组织方式。
2. 提交同步或异步报告任务。
3. 查看任务状态、步骤进度和已生成资产。
4. 审阅结构化章节、关键指标、图表、表格和结果摘要。
5. 需要进一步修改时，将已生成报告交给修订工作区。

### 输出

任务成功后，输出可能包括报告正文、图表、统计表、数据附件、JSON 证据、Markdown、HTML、PDF 和任务工件。实际输出由数据、报告路线、环境和质量门禁决定；不保证每次运行都得到全部格式。

### 前提与状态

报告任务编排默认可用。AI 辅助部分需要配置兼容提供方；正式管理报告仍必须通过下一节的完整发布门禁。

### 使用边界

- “生成了报告”可能指探索性、调试性或修订性材料；正式管理报告以发布门禁结果标识。
- 多表合并是否合理取决于共享键、粒度、时间范围和业务逻辑，系统建议用于辅助数据建模判断。
- 在质量门或证据链失败时，任务会反馈限制或生成调试材料；正式 PDF 仅在所有发布条件通过后生成。

### 对应职责模块

报告编排主要由 <code>orchestration_service.py</code>、<code>report_service.py</code>、<code>report_coordinator_service.py</code> 和 <code>job_graph_service.py</code> 承担；常用接口包括 <code>/api/datasets/{dataset_id}/smart-report</code>、<code>/api/datasets/{dataset_id}/smart-report-jobs</code>、<code>/api/report-jobs/{job_id}</code> 和 <code>/api/reports</code>。

## 8. 模块 06：AI 强制发布链与正式 PDF 门禁

### 解决的问题

防止未经字段理解、业务路由、指标规划、确定性计算和证据校验的内容被包装成正式管理报告。

### 入口

这是正式报告后端的强制流程。用户在 <code>/analysis</code> 提交正式报告任务后，相关服务负责执行和检查。

### 必经链路

<pre>
raw data
-> DataProfileService
-> AIFieldSemanticMapper
-> AIBusinessContextRouter
-> AIMetricDerivationPlanner
-> DeterministicMetricExecutor
-> EvidenceValidator
-> ReportBindingLayer
-> FormalPDFReleaseGate
-> management_report.pdf
</pre>

### 每个职责层做什么

| 层 | 负责内容 | 不允许做的事情 |
| --- | --- | --- |
| DataProfileService | 数据结构、字段、缺失、对象候选和工作表概览 | 仅凭结构直接定稿业务含义 |
| AIFieldSemanticMapper | 字段语义与候选业务角色的结构化映射，并写入 trace | 跳过 trace 或把自然语言猜测当作最终数值 |
| AIBusinessContextRouter | 根据数据、字段语义、用户目标选择业务路线 | 在正式发布中被跳过 |
| AIMetricDerivationPlanner | 区分直接指标、推导指标、代理指标、假设和不支持项 | 直接生成最终数值 |
| DeterministicMetricExecutor | 执行实际数值计算 | 使用模型文本代替计算 |
| EvidenceValidator | 检查计算输入、证据绑定和 schema 有效性 | 接受格式错误或无支撑的 AI 输出 |
| ReportBindingLayer | 将已验证的理解和计算绑定为报告结构 | 让渲染器重新理解字段或业务 |
| FormalPDFReleaseGate | 在发布前检查所有必经条件和质量分 | 缺少 trace、证据或分数不足时放行 |

### 输出

- 字段语义、业务路由、指标规划和使用门禁的持久化 trace。
- 已验证的指标、证据绑定和报告结构。
- 只有在门禁通过时才可发布的正式 <code>management_report.pdf</code>。

### 前提与状态

这是受门禁约束的能力。AI 强制步骤必须有可用配置、结构化输出和 schema 校验结果；最终数值必须由确定性执行器产生。

### 使用边界

- 缺少、无效或不一致的 AI trace 时，系统最多可以生成 <code>debug_report</code> 用于诊断，禁止发布正式 <code>management_report.pdf</code>。
- 最终质量分低于 90 时，禁止发布正式 PDF。
- 渲染器呈现已经绑定和验证的内容；字段理解、业务路由和指标推导由前置服务完成。
- Analysis Lab、通用智能报告和修订版本均受此门禁约束。

完整规则见 [AI 强制发布链架构](architecture_ai_mandatory_chain.md) 和 [正式报告可信机制](report-integrity.zh-CN.md)。

## 9. 模块 07：Analysis Lab 方法实验台

### 解决的问题

为分析师提供一个可探索、比较、复用和沉淀方法级证据的工作区，使方法选择和运行配置不被一次性报告任务吞没。

### 入口

<code>/lab</code>，主要为分析与方法负责人设计。

### 输入

- 数据集、工作表和字段画像。
- 目标、分组、标签、时间、特征、对象及图形字段角色。
- 派生指标的显示名、字段名、计算方式、来源字段和人工覆盖。
- 方法、运行实例、对象范围与统计选项。

### 核心操作

1. 完成“上传 -> 理解 -> 清洗 -> 派生 -> 可用”的数据准备流程。
2. 检查字段类型和缺失率，指定分析字段角色。
3. 定义或恢复派生指标建议，提交明确的计算口径。
4. 搜索方法目录，按方法族、来源、方法包或推荐结果筛选。
5. 为同一方法创建多个互不覆盖的运行实例，分别服务总体、重点对象或不同对比口径。
6. 配置全数据、字段绑定或重点对象三种数据范围，以及显著性、假设方向、聚类数、实验指标等选项。
7. 选择合并执行、单方法独立执行、批量独立执行或正式报告试跑。
8. 保存方法卡、追加运行、审阅结果与可重跑的报告部件。

### 输出

- 方法卡、字段绑定、运行实例和统计选项。
- 分析摘要、指标、图表、表格、文本和方法前置审计。
- 视本次结果而定的计算表、证据表、报告部件蓝图和资产清单。
- PDCA 状态和周期性改进信息（若对应服务在本次任务中运行）。

### 前提与状态

Analysis Lab 是默认可用的探索性工作区。其界面包含较广的方法目录，但实际可执行范围仍取决于数据、方法输入和本机运行环境。

### 使用边界

- Lab 提供探索性结果；正式管理报告仍需 AI trace、schema 校验、确定性数值、证据验证和质量门禁。
- 派生指标界面用于定义和提交口径；计算由后端确定性执行器完成。
- 重点对象筛选和手工输入值需要由使用者确认权限、取值格式和业务含义。

### 对应职责模块

主要后端职责包括 <code>auto_analysis_service.py</code>、<code>auto_analysis_field_service.py</code>、<code>auto_analysis_method_executor_service.py</code>、<code>auto_analysis_visual_service.py</code> 和 <code>analysis_lab_pdca_service.py</code>；常用接口包括 <code>/api/analysis/auto</code>、<code>/api/analysis/auto/methods</code>、<code>/api/lab/methods</code>、<code>/api/lab/method-cards</code>、<code>/api/lab/run</code> 和 <code>/api/lab/pdca/status</code>。

## 10. 模块 08：方法指南

### 解决的问题

帮助不熟悉分析平台的使用者从业务问题、变量类型和对象范围开始配置方法，结合研究目标选择合适路径。

### 入口

<code>/lab/method-guide</code>，也可以从 Lab 的方法编辑器打开。

### 核心内容

方法指南围绕四个实际操作主题组织：

1. **重点对象筛选**：何时需要只分析一组门店、客户、商品或时间段。
2. **字段角色绑定**：如何区分结果、解释、分组、对象、时间和图形输入字段。
3. **运行实例**：同一方法可保存多份配置，支持比较和复用。
4. **解释与建模输入字段**：如何把研究问题转换为方法需要的输入。

每个主题包含使用原因、操作步骤、样例、预期交付物和检查点。

### 输出

- 方法选择和配置的操作依据。
- 进入 Lab 后可复用的字段绑定、对象范围和运行实例思路。

### 前提与状态

默认可用；AI 密钥为可选配置。

### 使用边界

方法指南帮助用户理解操作，不替代业务判断、统计专业审阅、数据授权确认或因果研究设计。

## 11. 模块 09：报告资产、报告目录与下载文件

### 解决的问题

把一次运行生成的报告、图表、表格、结构化证据和辅助文件变成可查找、可复核、可续改的资产，供后续工作持续使用。

### 入口

- 正式分析任务完成后的结果区。
- 报告列表和报告详情。
- 修订中心 <code>/revision</code>。

### 核心操作

- 查看结构化章节、关键指标、图表和表格。
- 获取本次任务实际生成的文件和资产清单。
- 浏览历史报告，按关键词、数据集或业务画像筛选，按更新时间或生成时间排序。
- 将已有报告带入修订工作区继续处理。

### 输出

根据任务路径和执行结果，资产可能包括 Markdown、HTML、PDF、CSV、XLSX、JSON、图表、表格、报告部件、manifest、trace 和调试文件。具体格式和文件数量由任务路径和执行结果决定。

### 前提与状态

报告目录和基础浏览为默认可用。任务产物是否存在、是否可下载、是否包含某种格式，取决于本次执行是否完成以及所选路线是否满足条件。

### 使用边界

- 本地报告资产可能包含上传数据、业务描述或模型上下文；公开发布前需完成内容审查和脱敏。
- <code>/storage</code> 提供本地生成工件；公网文件分发由独立部署服务承担。
- 只有通过正式门禁的 <code>management_report.pdf</code> 才可称为正式管理报告。

## 12. 模块 10：报告修订工作台

### 解决的问题

将“请改一下这份报告”的零散意见转换为有会话、批注、附件、进度、差异和版本状态的后续交付流程。

### 入口

- 修订中心：<code>/revision</code>。
- 修订工作区：<code>/revision/workspace?report_id=...&session_id=...</code>。

### 输入

- 已生成并可浏览的报告。
- 自由文本修改意见或结构级、首页、图注等快捷修改提示。
- HTML/PDF 上的批注、附件和会话上下文。

### 核心操作

1. 在修订中心搜索、筛选和打开既有报告。
2. 创建新会话或恢复历史会话；运行时不可用时界面应进入只读查看，而不假装可以编辑。
3. 提交修改意见，并通过事件流或轮询查看进度。
4. 在 HTML/PDF 预览中刷新、缩放、翻页、整页查看、适宽查看或新窗口打开。
5. 使用浏览、画笔、矩形、高亮和橡皮工具添加或调整批注。
6. 查看生成产物、文件差异、附件、批注和本地已发布版本。
7. 在后端允许时查看或处理报告任务、CLI 管线、Codex 回合和恢复策略。

### 输出

- 修订会话、消息和事件记录。
- 批注与附件。
- 产物差异摘要/预览，以及本次实际生成的 Markdown、HTML、PDF 或调试文件。
- 满足工作区条件后的本地修订版本。

### 前提与状态

报告浏览和基础工作区能力可用。持续修改、自动续跑、Codex 相关恢复和本地发布动作会受到报告状态、运行时可用性、后端授权和验证结果限制。

### 使用边界

- “发布修订版”在报告工作台中创建本地版本；GitHub、网站和公开部署使用各自的发布流程。
- 系统依据错误、缺失输入和服务端策略展示可恢复状态与后续操作。部分失败需要修正输入或服务配置后重试。
- 修订流程不会绕过正式 <code>management_report.pdf</code> 的强制链与质量门禁。

### 对应职责模块

后端由 <code>report_agent_session_service.py</code> 等服务维护报告、会话、消息、事件流、附件、批注、文件差异与本地版本。接口分组包括 <code>/api/reports/{report_id}/agent-sessions</code>、<code>/api/report-agent-sessions/{session_id}/messages</code>、事件流、<code>diff</code>、<code>attachments</code>、<code>annotations</code> 和 <code>publish</code>。

## 13. 模块 11：可选运行时与本地扩展

### 解决的问题

为有明确运维权限和本机依赖条件的团队提供扩展入口，同时保持公共发布版的默认安全边界。

### 11.1 AI 提供方配置

**用途**：启用 AI 辅助字段语义、业务路由、指标规划、报告或修订增强。

**配置方式**：从 <code>.env.example</code> 创建本地 <code>.env</code>，按自己的提供方填写 <code>OPENAI_API_KEY</code>，并可按需设置 <code>OPENAI_BASE_URL</code>、<code>OPENAI_MODEL</code> 和 <code>OPENAI_REASONING_EFFORT</code>。

**边界**：密钥只保存在本地环境变量或 <code>.env</code>；GitHub 提交、截图和公开文本中不包含密钥。启用后，相关数据上下文可能发送给所选服务商。

### 11.2 R 工作流

**用途**：在具备本机 R 环境时执行可选的数据准备、统计、可视化、问卷分析和工件导出，并关联 R 产物。

**前提**：本机必须安装可用 R，并能找到 <code>Rscript</code>。

**状态和边界**：该能力需要独立配置，不列入便携版默认能力。未配置时界面会显示明确错误。

### 11.3 Codex Runtime

**用途**：在本地调用 Codex CLI 的健康检查、直接任务、异步任务、日志、取消、管线、阶段重试和进程管理能力。

**前提**：实际执行需要 <code>ASTERIA_CODEX_RUNTIME_ENABLED=1</code>、可用 Codex CLI 和受限工作区；通用 Runtime 任务、日志、取消和管线 API 还需要 <code>ASTERIA_ENABLE_CODEX_RUNTIME_API=1</code>。健康检查显示 CLI 可用性，任务执行授权由运行时配置确认。

**可见结果**：健康检查显示候选路径、解析路径、版本和错误；真实任务返回运行/任务 ID、状态、摘要、日志、文件变化、差异、工件或失败原因，具体字段取决于实际任务是否完成。

**状态和边界**：这是高权限本地扩展，运行于受控本机环境。公开互联网 API 和开箱即用云服务不在此能力范围。完整启动和排障顺序见 [本地扩展指南](local-extensions.zh-CN.md)。

### 11.4 外部 Skill、实验特性与本地报告 Agent Team

**用途**：在 Lab 中查看已挂载 Skill，导入或挂载经审阅的外部 Skill/插件包，运行 Feature Trial，导入本地 Report Agent Team，并在服务端许可时创建相应 Codex 任务。

**前提**：读取列表和目录提供扩展发现；安装、导入、挂载、卸载、删除、试验和团队运行需要 <code>ASTERIA_ENABLE_LOCAL_SKILL_INSTALLER=1</code>。Team Run 还需要已挂载团队、已选数据集和可运行的 Codex Runtime。

**可见结果**：Skill 可显示来源、挂载状态和包元数据；Feature Trial 产生就绪度、推荐动作、建议的 Lab 参数及 JSON/CSV/Markdown 工件；Team Run 返回任务 ID、状态和受控工作区路径。当前 Team 是导入角色 Markdown 后交给单个受控 Codex 任务的上下文，不承诺后端并行调度或自动完成正式报告。

**状态和边界**：第三方 Skill、团队配置和实验特性按可选扩展处理。导入前检查代码、许可证、依赖、网络行为和数据访问范围；包内声明的命令或 MCP 配置由使用者显式启用。完整操作见 [本地扩展指南](local-extensions.zh-CN.md)。

## 15. 模块 13：本地部署、隐私与安全

### 解决的问题

让用户能够启动项目、定位日志、保管密钥，并避免把原本设计为本机单用户使用的服务误公开到不安全的网络环境。

### 启动入口

| 发行方式 | 启动方式 | 前置条件 |
| --- | --- | --- |
| 源码版 | Windows 双击 <code>start-asteria.cmd</code> | Python 3.11+、Node.js 20.9+；首次安装需要可用网络 |
| Windows 便携版 | 解压发布资产后双击 <code>start-asteria.bat</code> | Windows 可写目录；无需预装 Python/Node.js |

启动器会检查前后端、准备依赖、启动本机服务，并将日志写入 <code>logs/launcher/</code>。默认前端为 <code>http://127.0.0.1:3000/analysis</code>，后端健康检查为 <code>http://127.0.0.1:8000/health</code>；端口冲突时应使用启动器打印的实际地址。

### 默认安全模型

- 回环地址、单用户、本机工作台。
- 浏览器不保存 API Key；密钥由服务器环境变量和本地 <code>.env</code> 管理。
- CORS 默认针对 <code>localhost/127.0.0.1</code>。
- Codex Runtime、Runtime API、非沙箱执行、本地 Skill/Team 安装和 Codex 搜索默认关闭。

### 使用边界

- 当前公开项目的默认部署未包含多用户认证、授权、租户隔离、上传安全治理和公网运维承诺。互联网或共享网络部署需要独立完成这些能力的设计与实施。
- 非本机部署需要独立设计身份认证、访问控制、数据隔离、密钥管理、审计、上传扫描、备份、监控和合规策略。
- 所有公开材料必须排除真实数据、报告、令牌、内部地址、日志和可识别客户的信息。

更多说明见 [快速开始](getting-started.zh-CN.md)、[部署边界](deployment-boundaries.md) 和 [安全策略](../SECURITY.md)。

| Analysis Lab | <code>/api/lab/methods</code>、<code>/api/lab/method-cards</code>、<code>/api/lab/run</code> | 管理方法、运行实例和实验结果 |
| 智能报告 | <code>/api/datasets/{dataset_id}/smart-report</code>、<code>/api/report-jobs/{job_id}</code>、<code>/api/reports</code> | 提交、跟踪和浏览报告任务 |
| 修订工作区 | <code>/api/reports/{report_id}/agent-sessions</code>、消息、事件、差异、附件、批注和发布接口 | 管理续改会话和报告版本 |
| 受控扩展 | <code>/api/runtime/codex-health</code>、<code>/api/lab/skills/*</code>、<code>/api/lab/report-agent-teams/*</code> | 仅在服务端明确启用时可用 |

## 18. 推荐工作流

### A. 基础本地分析

启动服务 -> 上传 CSV/XLSX -> 选择工作表 -> 检查画像 -> 写业务问题 -> 可选运行受控统计 -> 生成智能报告 -> 先审阅证据，再下载结果。

### B. 方法探索

进入 Lab -> 上传或选择数据 -> 检查字段画像 -> 配置字段角色和派生口径 -> 选择方法并建立运行实例 -> 单项、批量或合并执行 -> 审阅方法证据 -> 需要正式交付时进入正式报告链。

### C. 报告修订

进入修订中心 -> 搜索并打开已有报告 -> 创建或恢复会话 -> 提交修改要求或在 HTML/PDF 上批注 -> 审阅产物和差异 -> 仅在工作区条件满足时发布本地修订版。
